Cloud Manager
Azure Portal Reimagined

A production-grade web application built with ASP.NET Core 9 MVC that simulates the Microsoft Azure portal — complete with subscription management, Docker-backed virtual machines, real-time billing, role-based access control, and Entra ID user management.

ASP.NET Core 9 EF Core + SQL Server Docker + linuxserver/webtop JWT Cookie Auth BCrypt Password Hashing RBAC — 3 Roles Real-Time Cost Billing Entra ID Simulation Azure-Style UI Kubernetes + Cloudflare Tunnel
What we built

Key Features & Services

Cloud Manager delivers a complete cloud infrastructure management experience with enterprise-grade features implemented from scratch.

☁️

Azure Resource Hierarchy

Full 3-level hierarchy: Subscriptions → Resource Groups → Virtual Machines, mirroring the exact Azure model with cascade operations.

🔐

Stateless JWT tokens stored in HttpOnly, Secure, SameSite=Strict cookies. BCrypt password hashing. 8-hour session expiry with auto-redirect.

💰

👥

Entra ID Simulation

📋

Full audit trail of every VM operation — Create, Start, Stop, Delete, Restart — with timestamps stored in VmActivityLog for compliance.

// POST /Auth/Login
[HttpPost]
public async Task<IActionResult> Login(LoginDto dto)
{
    if (!ModelState.IsValid) return View(dto);

    var (success, token, error) = await _authService.LoginAsync(dto);

    if (!success) { ModelState.AddModelError("", error); return View(dto); }

    Response.Cookies.Append("jwt", token, new CookieOptions
    {
        HttpOnly = true,   // JS cannot read — XSS protection
        Secure   = true,   // HTTPS only
        SameSite = SameSiteMode.Strict,  // CSRF protection
        Expires  = DateTimeOffset.UtcNow.AddHours(8)
    });

    return RedirectToAction("Index", "Dashboard");
}

// POST /Auth/Register — auto-creates subscription + Owner role
[HttpPost]
public async Task<IActionResult> Register(RegisterDto dto)
{
    if (!ModelState.IsValid) return View(dto);
    var (success, error) = await _authService.RegisterAsync(dto);
    if (!success) { ModelState.AddModelError("", error); return View(dto); }
    TempData["Success"] = "Account created! Please login.";
    return RedirectToAction(nameof(Login));
}
💡 Auto-provisioning: On register, AuthService atomically creates: (1) User with Role="Owner", (2) a default Subscription, (3) a RoleAssignmt with RoleName="Owner" scoped to that subscription.

Controllers/DashboardController.csAggregation 
[Authorize]
public async Task<IActionResult> Index()
{
    var userId = GetUserId();
    var subs   = await _subRepo.GetByOwnerAsync(userId);

    var allRgs = new List<ResourceGroup>();
    var allVms = new List<VirtualMachine>();

    foreach (var sub in subs)
    {
        var rgs = await _rgRepo.GetBySubscriptionAsync(sub.SubscriptionId);
        allRgs.AddRange(rgs);
        foreach (var rg in rgs)
            allVms.AddRange(await _vmRepo.GetByResourceGroupAsync(rg.ResourceGroupId));
    }

    var model = new DashboardViewModel
    {
        TotalSubscriptions  = subs.Count,
        ActiveSubscriptions = subs.Count(s => s.Status == "Active"),
        TotalResourceGroups = allRgs.Count,
        TotalVMs            = allVms.Count,
        RunningVMs          = allVms.Count(v => v.Status == "Running"),
        StoppedVMs          = allVms.Count(v => v.Status == "Stopped"),
        RecentVMs           = allVms.Take(5).ToList()
    };
    return View(model);
}

// Only Owner and Contributor can create
[HttpPost]
public async Task<IActionResult> Create(CreateResourceGroupDto dto)
{
    var role = await GetUserRoleOnSubscription(dto.SubscriptionId);
    if (role == "Reader") return Forbid();

    var (success, error) = await _rgService.CreateAsync(dto);
    if (!success) { ModelState.AddModelError("", error); return View(dto); }

    TempData["Success"] = $"Resource Group '{dto.Name}' created.";
    return RedirectToAction(nameof(Index), new { subscriptionId = dto.SubscriptionId });
}

// Only Owner can delete
[HttpPost]
public async Task<IActionResult> Delete(int id, int subscriptionId)
{
    if (await GetUserRoleOnSubscription(subscriptionId) != "Owner") return Forbid();
    var (success, error) = await _rgService.DeleteAsync(GetUserId(), id);
    TempData[success ? "Success" : "Error"] = success ? "Resource Group deleted." : error;
    return RedirectToAction(nameof(Index), new { subscriptionId });
}

// Role resolution: walks RG → Subscription to find user's effective role
private async Task<string?> GetUserRoleOnSubscription(int subscriptionId)
{
    int userId = GetUserId();
    var sub = await _subRepo.GetByIdAsync(subscriptionId);
    if (sub?.UserId == userId) return "Owner";  // subscription owner = always Owner
    var roles = await _roleRepo.GetByUserAsync(userId);
    return roles.FirstOrDefault(r =>
        r.ScopeType == "Subscription" && r.ScopeId == subscriptionId)?.RoleName;
}

// Restart — stop → delay → start (with role check)
[HttpPost]
public async Task<IActionResult> Restart(int id, int resourceGroupId)
{

// GET /Cost/BudgetStatus — JSON polled every 30s by the subscription banner
public async Task<IActionResult> BudgetStatus(int subscriptionId)
{
    var status = await _cost.GetBudgetStatusAsync(subscriptionId);
    return Json(new {
        spent        = status.SpentThisMonth.ToString("F2"),
        limit        = status.BudgetLimit.ToString("F2"),
        isOverWarning= status.IsOverWarning,
        isOverBudget = status.IsOverBudget,
        pctUsed      = status.BudgetLimit > 0
            ? Math.Round((double)(status.SpentThisMonth / status.BudgetLimit * 100), 1)
            : 0
    });
}

Module 08

Role-Based Access Control

Three roles scoped per Subscription. Every controller enforces role checks before executing any write operation. The role hierarchy walks up from VM → Resource Group → Subscription to find the user's effective role.

👑 Owner
Full control. Create, modify, delete everything. Assign/revoke IAM roles. The only role that can delete VMs and Resource Groups.
🔧 Contributor
Create and operate resources. Start/Stop VMs. Create Resource Groups. Cannot delete or assign IAM roles.
👁 Reader
View-only. All mutating actions return 403 Forbidden. Can view VMs and the browser console.

8.1  Core Kubernetes Concepts Used

Kubernetes organises workloads through a hierarchy of abstractions. The table below explains each concept and how it maps to this project.

ConceptWhat It IsHow It's Used Here
Cluster A group of machines (nodes) managed by Kubernetes as one unit — mirrors a real cloud server environment. One local cluster created to host the entire Cloud Manager application.
Node A single machine inside the cluster — can be physical or virtual. One node acts as both control plane and worker.
Pod The smallest deployable unit in Kubernetes. A Pod wraps one or more containers and runs on a Node. Not the same as a Docker container — a Pod is a K8s abstraction over containers. Two Pods: one for the ASP.NET Core app, one for SQL Server.
Deployment A K8s resource that declares the desired state for Pods — which image to run, how many replicas, restart policy, etc. One Deployment YAML for the app Pod, one for the SQL Server Pod.
Service A stable network endpoint that exposes Pods. Unlike a Pod's IP (which changes on restart), a Service IP is fixed and persistent. A Service YAML is required so the app can always reach SQL Server at a fixed address, regardless of Pod restarts.
💡 Why a Service alongside the Deployment? If the app relied directly on the SQL Server Pod IP, that IP would change every time the Pod restarted. The Service provides a stable DNS name and IP that never changes — essential for reliable database connectivity.

8.2  Configuration Files (YAML)

8.3  Pod Architecture

The cluster contains two Pods. The ASP.NET Core MVC application is kept as a single Pod because the MVC pattern compiles UI (Razor Views) and backend logic together — they cannot be split into separate deployable units.

PodContainer ImagePortReason for Separation
App Pod cloudmanager:latest 8080 ASP.NET Core MVC — UI and logic are inseparable in the MVC pattern.
SQL Server Pod mssql/server:2022 1433 Database runs in its own isolated Pod with a stable Service for connection string.
💡 Why not split the app into two Pods (UI + API)? Cloud Manager uses ASP.NET Core MVC, where Razor Views are server-rendered inside the same process as the controllers and services. There is no separate frontend bundle — splitting is not applicable to the MVC pattern.

8.5  Public Access via Cloudflare Tunnel

To expose the application publicly — without a static IP, without opening firewall ports, and accessible from any network — a Cloudflare Tunnel is established between the Kubernetes cluster and Cloudflare's edge network.

#StepDetail
1 Tunnel Created cloudflared is run in the terminal, creating an outbound-only encrypted tunnel from the local machine to Cloudflare's edge — no inbound ports needed.
2 URL Issued Cloudflare assigns a public HTTPS URL (e.g. https://abc123.trycloudflare.com). Anyone on any network can open this URL.
3 Traffic Routed Requests hit Cloudflare's edge → flow through the tunnel → reach the K8s Service → reach the App Pod.
Terminal (CMD)Cloudflare Tunnel — expose Kubernetes app publicly
# Run cloudflared pointing at the local Kubernetes service URL
cloudflared tunnel --url http://127.0.0.1:54321

# Cloudflare will output a public URL such as:
https://random-name.trycloudflare.com
#
# Anyone on any network can now access Cloud Manager at this URL.
✅ Key benefit: No public IP required, no firewall rules, no port forwarding. The tunnel is outbound-only from the local machine — Cloudflare handles all incoming traffic securely at the edge.
